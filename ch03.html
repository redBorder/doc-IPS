3.1. IPS/IDS sensors configuration

To configure the IPS/IDS sensors, it is necessary to access the redborder web platform by inserting the management IP assigned to the Manager in the address bar of your browser. 
https://IP-manager-address

Note

It is possible to log in to the platform using the default account: admin or admin@redborder.com with the password used during the Manager installation process.

Warning

The management web will not work if you try to access it through an unencrypted web.
There are three ways to access the edition of the IPS/IDS sensors for the configuration of their operating parameters:
  • Tree view: in the tree view, the user can see the IPS/IDS sensors that have been created, accompanied by an identification icon. Click on the sensor name or press the "Edit" button to access to the sensor configuration view
    Tree view: edit IPS/IDS sensor
  • Access Points view: the Access Points view show a list of APs defined on the network. Click on an Access Point to access the interface configuration for general sensor parameters.
    Access Points view: edit IPS/IDS sensor
  • IPS specific view: on the View there are specific views for each type of sensor. Select IPS sensors to access the alphabetical list of IPS sensors. Click the "Edit" icon.
    IPS Sensors view: IPS/IDS

3.1.1. Edit IPS/IDS settings overview

In all the cases described, the user accesses to a basic parameters configuration view.
The configuration form fields vary depending on whether you are editing a sensor, group of segments or binding. This step is very important since one configuration or another will determine the sensor's operating mode depending on whether it is installed in one scenario or another. These options are defined in the section IPS/IDS configuration.
The fields displayed on the IPS sensor configuration form are listed below:
  • General: displays general sensor information. The only editable information in this section is the description. [1]
  • IPS/IDS Settings

  • Network routes
  • Location
  • IP Variables

  • Groups
Editable information for IDS/IPS sensor configuration
In the upper right corner of the screen we find the following options:
  • Stop Auto-apply
    This button prevents Chef from automatically applying the changes (it refreshes and applies every 5 minutes).
    The system will open a confirmation window to make sure that this is the action the user wants to perform.
  • Options-> Recreate Private Key
    This option regenerates the synchronization private key of the IPS sensor with the Manager. If you have Administrator role in redborder, You will also find this option in the user editing interface in Tools -> Users.
    Once said restoration has been confirmed, the user must enter the text that is shown in the modal window in the IPS in the following path:: /etc/chef/client.pem

Warning

The action of restoring the private key is very sensitive since it can leave the sensor without communication with the Manager. Therefore, the system opens a confirmation window before performing any action. Make sure you are sure you want to restore the security key before confirming it.

3.1.2. IPS/IDS Settings

In this section we find the options available to configure the IPS/IDS sensors and the behavior of Snort and PF_RING.
Editable fields for IPS/IDS (IPS Settings)
Sensor mode
Default sensor mode: in this box will appear the mode that has been inherited from higher levels. In this case, the word INHERITED will appear followed, in parentheses, by the way it has been inherited.
In this field, the user can choose between 4 different options:
  1. IDS forwarding: traffic is immediately forwarded by the sensor; then a copy of it is analyzed. In practice, it produces the same effect as having a TAP before the sensor.
  2. IDS span: only the traffic is analyzed. No forward is made.
  3. IPS test mode: the traffic is analyzed and subsequently forwarded. This mode is recommended for simulations of blocking attacker traffic.
  4. IPS: this is the most common option in production environments. Snort analyzes the traffic and passes or blocks the traffic depending on the actions set on the rules included in the sensor signature policy.

Important

The sensor must be configured in IPS Test or IPS mode for the basic scenario of installing redborder IPS.
Next, the user can edit a series of parameters related to the behavior of PF_RING and Snort:
PF_RING Watermark
Indicates the minimum number of incoming packets for the poll function.
  • Low watermark value (1): reduces the latency of the poll function, decreases the throughtput, but may increase the number of calls to the poll function.
  • High watermark value (should never exceed 50% of the ring size or else the PF_RING kernel module will increase this value): reduce the number of poll calls, but slightly increases packet latency.
PF_RING Clustermode
Split incoming traffic into multiple rings. Especially suitable for distributing the load by allowing the same interface to be read from several Snort programs.
Indicates the number of parameters used to determine the hashing and balancing of the traffic between the different Snort processes. Depending on the number of elements that are added to the tuple, the traffic will be distributed in a different way:
  • 2 tuple flow: Source IP+Destination IP
  • 4 tuple flow: Source IP:Port_src+Destination IP:Port_dst
  • 5 tuple flow: Source IP:Port_src+Destination IP:Port_dst+protocol
  • 6 tuple flow: Source IP:Port_src+Destination IP:Port_dst+protocol+Vlan
Bypass controlled by Snort
When the machine starts Snort can automatically take control of the bypasses. This action will allow the system to switch off the bypass after a boot process.
Threshold Count
Maximum number of occurrences in a given time range for an event to be sent.
Threshold Seconds
Minimum time (in seconds) that has to pass for an event to be sent.

Note

In scenarios where there is an overload of traffic, it is possible make a configuration that allows connectivity over packet analysis.
The options of Best effort Configuration and Software Bypass Configuration are used to prioritize forwarding information over analyzing it.
Best effort Configuration
This setting only applies in IDS mode.
By enabling this option, traffic is copied to an internal work queue and then forwarded. Only if there are no more packets to be forward, the packets in the internal queue will be processed. If the internal queue is full, the packets will not be analyzed. This increases the reliability of the bypass, but decreases the performance of the analysis.
Software Bypass Configuration
This setting only applies in IPS mode.
The activation of this option implies that, if the occupation of the interface ring is high, IPS will forward unanalyzed packets until occupancy falls below some other predetermined value.

Important

IPS bypass configuration is done at the software level. To determine the configuration of the bypasses at the hardware level, refer to the section Segments.

3.1.10. Groups

Group: it is a set of instances associated with a specific processor that read from one or more segments in a specific way.
Instance: it is an association of a group of segments to a processor or processors. In case we have created more than one instance for the same group, it is necessary to have the same number of bindings configured as instances.
Binding it is a partition of the traffic by network or VLAN. It is a "child" of a group of segments.
The user can assign a policy at three levels: sensor, group of segments or bindings.
This increases the flexibility associated with the software, since, unlike the previous version in which a policy had to be fully applied for a sensor (regardless of the segments created), allows the formation of groups of independent segments and bindings.

Note

The inheritance system is established from the higher levels and will apply to the lower levels that are derived from it.
Configuration parameters and policies of higher levels affect those "children" that are derived or depend on them. In case the configuration of said children is not edited, the system will assign by default the policies, modes and configurations assigned to the "parent" level.
Segments groups
In this section you can configure the segment groups:
  • Name: edit group name
  • Sensor mode: selection of the mode to be applied at the group level (sensor mode is inherited by default)
  • Segments: indicates the segment(s) included in the group
  • CPUs: assignment of the processor(s) from which the group of segments is to be read.

Important

When there is more than one group of segments, it is recommended that different processors or CPUs be assigned to each group to distribute the processes and on that way balance the load of each processor.

3.1.10.1. Conditions for creating segment groups

  • It is not possible for two groups to share the same segments.
  • Two instances of the same group cannot have different modes (Sensor mode).
  • IPS/IDS sensors: they can only be derived from their own groups and bindings.
  • Groups and links can be moved and make them domain dependent. However, these can only be returned to the IPS sensor they belonged to before being moved.

3.1.10.2. Add/Edit Segment Group

To add a new group, simply select the option +Add Group.
To edit an existing group, click on the "Edit" icon.
In both cases, the user will access the configuration interface at the group level. In it you can modify the following information:
  • Name: edit group name
  • Description: add a description to the segment group.
  • IPS/IDS Settings: configuring Snort parameters at the group level.
  • IP Variables: list of values ​​for the Snort IP variables at the group level. Some Snort variables must be modified (edited) to suit the user's network infrastructure so that Snort can recognize them. Those that are inherited from a higher level will appear with a check in the Inherited box.
  • Port Variables: This section shows the values ​​for Snort ports variables at the group level. The values ​​of these variables can also be edited in exceptional cases where non-conventional ports have been defined. In this way, Snort will be able to recognize these variables. Those that are inherited from a higher level will appear marked as Inherited.
  • Bindings: bindings are partitions of traffic across networks or VLANs. They act as "children" of the group at the level of inheritance and application of policies. To add a new binding click on +Add binding and enter the name.

Note

Remember that the segment group will display the settings it has inherited from the sensor. This is the one that will be shown by default in the editing screen at the group level.
Add/edit segment group
Delete a segment group
A group can be deleted at any time using the "Delete" icon. The system will display a confirmation message.

Important

In case that a group had a series of policies assigned, when the group is deleted, these will no longer be applied to the segments that belonged to the group. The segments will be free for integration into another group.

Add/edit binding configuration
There are two ways to add or edit a binding:
  1. IPS/IDS sensor configuration view: Groups -> Edit icon -> Bindings-> + Add binding/Edit icon
  2. IPS/IDS sensor configuration view: + Add Group -> Bindings -> + Add binding/Edit icon
When we add a new binding the system prompts us for a name.
Segments groups -> Bindings

Note

For a binding to be taken into account, at least one Vlan must be assigned to it.
Edit binding configuration
The editing fields for the binding level configuration are the following:
  • General: Name and description.
  • IPS/IDS Settings
  • IP Variables
  • Port Variables
To ensure that modifications made to an existing binding will be applied correctly, click on Update

Warning

If the IPS/IDS Settings, IP Variables and Port Variables fields are not modified, the binding will inherit the configuration of the group of segments on which it depends.
